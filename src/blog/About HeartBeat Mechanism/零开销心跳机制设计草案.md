# 零开销心跳机制设计草案

> 一种基于超时续命的极简网络连接保活方案

---

## 1. 设计动机

传统心跳机制存在的问题：

| 问题 | 说明 |
|------|------|
| **周期性遍历** | 服务器需要定时器轮询所有连接，检查心跳状态，O(N) 复杂度 |
| **双向心跳开销** | 客户端发心跳 → 服务器回复 ACK，产生不必要的往返流量 |
| **空闲连接也有开销** | 即使业务繁忙、连接活跃，心跳包仍在定时发送 |
| **设计复杂度** | 需要维护心跳定时器、超时检测、重连策略等状态机 |

**核心目标**：在连接活跃时实现 **零心跳开销**，仅在静默期以最小代价保活。

---

## 2. 核心思想

```
传统模型:  Client --[heartbeat]--> Server --[ack]--> Client  (周期性，无条件)
本方案:    Client --[heartbeat]--> Server (丢弃)              (仅静默期，单向)
```

### 2.1 三个关键原则

1. **服务器只有「续命」，没有「心跳」**
   - 服务器不主动发送也不回复任何心跳帧
   - 服务器仅维护每个 socket 的超时时间
   - 任何来自客户端的数据（业务数据或心跳帧）都会自动为该 socket 续命

2. **客户端只在静默期才发送心跳**
   - 客户端维护一个比服务器更短的超时时间
   - 当客户端在此超时周期内没有发送任何业务数据时，才触发心跳帧
   - 业务数据活跃期间，心跳永远不会触发

3. **心跳帧是「哑」帧——发送即成功**
   - 服务器收到心跳帧后直接丢弃，不回复、不解析、不做任何操作
   - 客户端只要 `send()` 成功（TCP 层确认），就知道连接仍然存活
   - 无需等待服务器回复来确认保活

## 3. 超时模型

### 3.1 时间参数定义

```
设:
  T_server = 服务器 socket 超时时间（如 60s）
  T_client = 客户端静默超时时间（如 45s）

约束:
  T_client < T_server

安全余量:
  T_margin = T_server - T_client（如 15s）
```

### 3.2 时序图

#### Case 1: 业务活跃期（零心跳开销）

```
Client                              Server
  |                                   |
  |---[业务数据]--------------------->|  t=0s,  续命 → deadline=60s
  |<--[业务响应]----------------------|
  |---[业务数据]--------------------->|  t=10s, 续命 → deadline=70s
  |<--[业务响应]----------------------|
  |---[业务数据]--------------------->|  t=30s, 续命 → deadline=90s
  |                                   |
  ↑ 心跳永未触发，0 开销              |
```

#### Case 2: 静默期保活（最小开销）

```
Client                              Server
  |                                   |
  |---[最后一次业务数据]------------->|  t=0s,   续命 → deadline=60s
  |                                   |
  | (静默 45s，客户端超时触发)        |
  |                                   |
  |---[心跳帧, 空payload]----------->|  t=45s,  续命 → deadline=105s
  |                                   |  (服务器丢弃心跳帧，不回复)
  |                                   |
  | (客户端 send 成功 → 保活确认)     |
  | (重置客户端超时计时器)            |
  |                                   |
```

#### Case 3: 连接失效（优雅断开）

```
Client                              Server
  |                                   |
  |---[最后一次业务数据]------------->|  t=0s,   续命 → deadline=60s
  |                                   |
  |         (网络中断)                |
  |                                   |
  |---[心跳帧]-------X               |  t=45s,  send 失败
  |                                   |
  | (客户端检测到发送失败)            |  t=60s,  服务器超时，关闭 socket
  | (触发重连逻辑)                    |
  |                                   |
```

## 4. 架构设计

### 4.1 服务器端

```
┌─────────────────────────────────────────────────┐
│                   Server                         │
│                                                  │
│  ┌───────────────┐    ┌──────────────────────┐  │
│  │  epoll/io_uring│    │  Socket 超时管理器   │  │
│  │  事件循环      │───>│  (内核级/时间轮)      │  │
│  └───────┬───────┘    └──────────┬───────────┘  │
│          │                       │               │
│          │ 任意数据到达           │ 超时触发      │
│          ▼                       ▼               │
│  ┌───────────────┐    ┌──────────────────────┐  │
│  │ 续命: 重置该   │    │  关闭 socket         │  │
│  │ socket deadline│    │  释放资源             │  │
│  └───────────────┘    └──────────────────────┘  │
│                                                  │
│  注: 无心跳解析逻辑，无心跳回复逻辑             │
└─────────────────────────────────────────────────┘
```

**服务器核心行为**：

- **数据到达**（无论业务数据还是心跳帧）→ 重置该 socket 的超时 deadline
- **超时触发** → 直接关闭 socket，释放所有关联资源
- 心跳帧在「续命」之后即被丢弃，不进入业务逻辑层

### 4.2 客户端

```
┌──────────────────────────────────────────────────┐
│                   Client                          │
│                                                   │
│  ┌─────────────┐     ┌────────────────────────┐  │
│  │ 业务逻辑层  │     │  静默超时检测器         │  │
│  │             │     │  (T_client 定时器)       │  │
│  └──────┬──────┘     └───────────┬────────────┘  │
│         │                        │                │
│         │ send 业务数据          │ 超时未发送数据 │
│         │ → 重置定时器           │                │
│         ▼                        ▼                │
│  ┌─────────────┐     ┌────────────────────────┐  │
│  │   TCP Send  │     │  发送心跳帧(空payload) │  │
│  │             │     │  → send 成功: 保活确认  │  │
│  │             │     │  → send 失败: 触发重连  │  │
│  └─────────────┘     └────────────────────────┘  │
│                                                   │
└──────────────────────────────────────────────────┘
```

**客户端核心行为**：

- 每次 `send()` 业务数据时，重置静默超时定时器
- 定时器到期（静默 T_client 秒）→ 发送心跳帧
- 心跳帧 `send()` 成功 → 保活成功，重置定时器
- 心跳帧 `send()` 失败 → 连接可能断开，进入重连流程

## 5. 基于 Asio 的实现要点

### 5.1 服务器端伪代码

```cpp
class Connection : public std::enable_shared_from_this<Connection> {
    asio::ip::tcp::socket socket_;
    asio::steady_timer    deadline_timer_;
    
    static constexpr auto kServerTimeout = std::chrono::seconds(60);

    void start() {
        renew_deadline();
        do_read();
    }

    void renew_deadline() {
        deadline_timer_.expires_after(kServerTimeout);
        deadline_timer_.async_wait(
            [self = shared_from_this()](const asio::error_code& ec) {
                if (!ec) {
                    // 超时，直接关闭连接
                    self->socket_.close();
                }
            });
    }

    void do_read() {
        socket_.async_read_some(asio::buffer(buf_),
            [self = shared_from_this()](asio::error_code ec, std::size_t len) {
                if (ec) return; // 连接断开

                // 任何数据到达 → 无条件续命
                self->renew_deadline();

                if (self->is_heartbeat(buf_, len)) {
                    // 心跳帧：续命已完成，直接丢弃，不回复
                } else {
                    // 业务数据：交给上层处理
                    self->dispatch_business_data(buf_, len);
                }

                self->do_read();
            });
    }

    bool is_heartbeat(const char* data, std::size_t len) {
        // 心跳帧识别：可以是固定 magic byte 或空 payload
        return len == sizeof(kHeartbeatMagic) 
            && std::memcmp(data, &kHeartbeatMagic, len) == 0;
    }
};
```

### 5.2 客户端伪代码

```cpp
class ClientConnection {
    asio::ip::tcp::socket socket_;
    asio::steady_timer    silence_timer_;
    
    static constexpr auto kClientTimeout = std::chrono::seconds(45);

    void reset_silence_timer() {
        silence_timer_.expires_after(kClientTimeout);
        silence_timer_.async_wait(
            [this](const asio::error_code& ec) {
                if (!ec) {
                    send_heartbeat();
                }
            });
    }

    void send_business_data(const std::vector<char>& data) {
        asio::async_write(socket_, asio::buffer(data),
            [this](asio::error_code ec, std::size_t) {
                if (!ec) {
                    // 发送成功 → 重置静默计时器（业务数据已经替代心跳的作用）
                    reset_silence_timer();
                }
            });
    }

    void send_heartbeat() {
        asio::async_write(socket_, asio::buffer(&kHeartbeatMagic, sizeof(kHeartbeatMagic)),
            [this](asio::error_code ec, std::size_t) {
                if (!ec) {
                    // send 成功 → TCP 层保活确认，重置计时器
                    reset_silence_timer();
                } else {
                    // send 失败 → 连接可能已断，触发重连
                    on_connection_lost();
                }
            });
    }
};
```

### 5.3 心跳帧协议设计

```
最简方案 — 固定 Magic Byte:

  ┌────────────────────┐
  │  0xHB  (1 byte)    │   // HeartBeat magic, 服务器据此识别并丢弃
  └────────────────────┘

或 带帧头方案（与业务协议统一）:

  ┌──────────┬──────────┬──────────┐
  │ FrameType│ Length   │ Payload  │
  │ 0x00(HB) │ 0x0000   │  (空)    │
  └──────────┴──────────┴──────────┘
```

## 6. 与传统方案对比

| 维度 | 传统双向心跳 | 本方案（超时续命） |
|------|-------------|-------------------|
| 服务器心跳逻辑 | 定时遍历所有连接 O(N) | 无，仅 epoll/io_uring 事件驱动 |
| 心跳网络开销 | 每个连接周期性双向包 | 仅静默期单向，活跃期为零 |
| 服务器回复心跳 | 需要回复 ACK/Pong | 不回复，直接丢弃 |
| 活跃连接开销 | 仍有心跳包发送 | **零开销** |
| 定时器数量 | 全局心跳定时器 + 每连接检测 | 每连接一个 deadline timer（内核管理） |
| 实现复杂度 | 心跳状态机 + 超时检测 + ACK 处理 | 极简：超时关闭 / 数据续命 |
| 保活确认 | 依赖服务器 ACK | 依赖 TCP `send()` 返回值 |
| 万级连接场景 | 心跳风暴，CPU 遍历开销大 | 活跃连接无额外开销 |

## 7. 边界情况与应对策略

### 7.1 半开连接（Half-Open）

**场景**：网络中间设备（NAT/防火墙）静默丢弃连接，双方均不知情。

**应对**：
- T_client 远小于 NAT 超时时间（通常 NAT 超时 > 120s），保证客户端在 NAT 丢弃前发出心跳
- 客户端 `send()` 成功仅表示数据进入内核发送缓冲区，极端情况下可配合 `TCP_USER_TIMEOUT`（Linux）做内核级超时检测

### 7.2 服务器端 timer 管理效率

**方案选择**：
- **时间轮（Timing Wheel）**：O(1) 插入/删除，适合大量连接且超时时间相近的场景
- **最小堆（Min-Heap）**：O(log N) 操作，Asio `steady_timer` 默认实现
- **内核级 `epoll_wait` timeout**：直接利用 epoll 超时参数，零用户态开销

### 7.3 心跳帧与业务帧区分

建议在协议帧头使用 1 字节 `FrameType` 字段区分：
- 服务器解析帧头后，若为心跳类型则跳过后续处理
- 开销：每帧额外 1 字节判断，可忽略不计

### 7.4 客户端 `send()` 成功 ≠ 对端收到

TCP `send()` 成功仅代表数据写入内核缓冲区。真正的投递保证依赖 TCP 重传机制。

**加固方案**：
- Linux: 设置 `TCP_USER_TIMEOUT` socket option，内核在指定时间内未收到 ACK 则报告错误
- 应用层: 可选的「challenge-response」模式作为降级策略（不推荐常态使用）

## 8. 性能分析

### 8.1 理论开销

设 N 个并发连接，活跃比例为 α（0 ≤ α ≤ 1）：

```
传统方案每周期开销:
  心跳包数 = 2N（每连接一发一收）
  遍历开销 = O(N)

本方案每周期开销:
  心跳包数 = (1 - α) × N（仅静默连接，且单向）
  遍历开销 = 0（事件驱动，无主动遍历）
```

当 α → 1（高活跃场景），本方案心跳开销趋近于 **零**。

### 8.2 万级连接场景估算

| 场景 | 连接数 | 活跃率 | 传统方案心跳/s | 本方案心跳/s |
|------|--------|--------|---------------|-------------|
| 即时通讯 | 10,000 | 30% | 20,000 | 7,000 |
| 游戏服务器 | 10,000 | 80% | 20,000 | 2,000 |
| IoT 网关 | 100,000 | 5% | 200,000 | 95,000 |
| 推送服务 | 100,000 | 1% | 200,000 | 99,000 |

> 注：本方案数值为单向包，传统方案为双向包（实际带宽差距更大）。
> 且本方案服务器侧无需遍历，CPU 节省更为显著。

## 9. 总结

本方案的核心哲学：

> **不做无用功** —— 活跃连接的业务数据本身就是最好的「心跳」。

设计要点回顾：

1. **服务器极简**：只管超时关闭，数据到达即续命，不区分心跳与业务
2. **客户端自治**：自行判断静默期，自行发送保活帧，自行确认成功
3. **零回复**：服务器不回复心跳，减少一半网络开销
4. **零遍历**：依赖 epoll/io_uring 事件驱动，无定时轮询
5. **零活跃开销**：业务活跃期间心跳永不触发
