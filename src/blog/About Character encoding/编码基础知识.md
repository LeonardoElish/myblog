# 什么是编码
**编码**就是用一定的规则，将字符（如汉字、英文、符号、表情等）转换成计算机可以存储和传输的字节（数字）序列。不同的编码规则，对同一个字符的表示方式不同。例如："中" 这个字在 GBK 编码和 UTF-8 编码下对应的字节就不一样。

---
##  常见编码方式

- **ASCII**：最基础，英文单字节编码（0~127）。
	- 7位编码，表示128个字符（基础英文和控制符）。
	- 兼容性好，但无法表示非英语字符（如中文）。
- **GB2312/GBK/GB18030**：中国大陆常用的汉字编码（多字节）。[[GB2312、GBK 和 GB18030 ASCII]]
	- **GB2312**：中国大陆早期的汉字编码，支持约7000汉字。
	- **GBK**：GB2312的扩展，支持更多汉字及符号（兼容GB2312）。    
	- **GB18030**：国家标准，支持所有中文字符和Unicode扩展，向下兼容GBK。
- **BIG5**：台湾、香港常用的繁体中文编码。
- **ISO-8859-1(Latin-1 )**：西欧语言编码。
	-  单字节编码，支持西欧语言和符号（共256字符）。
	-  不能表示中文、日文等东亚字符。
- **Unicode**：全球字符统一编码标准（只是编号，不规定存储方式）。
- **UTF-8**：变长编码，英文1字节，中文3字节，兼容ASCII，互联网和跨平台首选。
- **UTF-16**：2/4字节，Windows、Java内部常用
	- Windows和Java内部常用。
	- 小端（LE）与大端（BE）区别。
- **UTF-32**：定长4字节，少见。
	- 定长编码（4字节），空间大但处理简单。

## 什么时候需要“转码”？

### 需要转码的典型场景
- **不同系统/软件/文件间数据交互**  
  例：Windows默认GBK，Linux服务器默认UTF-8，文件从Windows拷到Linux可能出现乱码。
- **网络传输、API交互**  
  例：HTTP接口一般约定UTF-8，否则多语言内容会乱码。
- **数据库、日志、文件存储**  
  例：数据库设置编码为UTF-8，插入GBK数据会乱码。
- **显示与输入**  
  例：控制台、网页等显示内容，字体/环境编码不一致会乱码。

### 不需要转码的场景
- **编码一致**：写入、存储、读取、显示全部用同一编码（如全用UTF-8）。
- **仅英文**：用ASCII编码，绝大多数编码兼容ASCII，不会乱码。
---
##  常见默认编码

- **Linux、MacOS**：默认UTF-8。
- **Windows控制台/记事本**：历史上默认GBK/ANSI，Win10以后逐步支持UTF-8。
- **网页HTML**：一般 `<meta charset="utf-8">`，默认UTF-8。
- **Java/C#/Python等主流编程语言**：新项目模板通常默认UTF-8。
- **C++/C**：标准库不限定编码，依赖操作系统和编译器设置。
---

##  代码文件中的字符串编码和什么有关？

**代码文件中的字符串字面量（如 "你好"）的编码，主要取决于文件本身保存时用的编码。**

### 具体影响因素
- **文件保存编码**：如你用 VSCode 保存时选择 UTF-8，文件内容就是 UTF-8。
- **编译器/解释器设置**：如 GCC/MSVC 编译 C/C++ 代码时，源文件编码与编译选项（-finput-charset，-fexec-charset）相关。
- **语言标准**：如 Python 3 默认源文件 UTF-8，C++ 标准不限定，Java 也推荐 UTF-8。
- **字符串类型**：
  - 普通字符串（如 C++ 的 `std::string`）：其实只是字节序列，解释时才知道编码。
  - 宽字符串（如 `std::wstring`）：与操作系统有关，Windows多为UTF-16，Linux多为UTF-32。
  - C++11 引入 `u8""`、`u""`、`U""` 等前缀，明确指定字符串字面量的编码（UTF-8、UTF-16、UTF-32）。

---

##  建议

- **新项目建议统一用 UTF-8**，避免跨平台乱码。
- **存储和网络接口统一用 UTF-8**。
- **遇到乱码优先检查源文件、终端、数据库等各环节的编码设置是否一致。**
- **跨语言/平台时，显示和处理前后注意转码。**
