场景如下:

> 假设有Stock类，代表一只股票的价格。每一只股票有一个唯一的字符串标识(key)，
> 比如 Google 的 key是"NASDAQ:GOOG”，IBM是"NYSE:IBM”。
>
> Stock 对象是个主动对象，它能不断获取新价格。为了节省系统资源，同一个程序里边每一只出现的股票只有一个Stock对象，如果多处用到同一只股票，那么Stock对象应该被共享。如果某一只股票没有再在任何地方用到，其对应的Stock对象应该析构，以释放资源，这隐含了“引用计数”。

按照上述需求,设计一个对象池StockFactory

代码见: [StockFactory](./ch1/StockFactory.cpp)

## version1

```cpp
// version 1: questionsable code
class StockFactory : boost::nocopyable
{
public: 
	shared_ptr<Stock> get(const std::string& key);
private:
	mutable MutexLock 						m_mutex;
	std::map<std::string,shared_ptr<Stock>> m_stocks;
};

```

它存在以下问题:

1. 强引用持有: `m_stocks`存储的是 `std::shared_ptr<Stock>`。`shared_ptr` 是一个**强引用**，它会阻止对象被销毁。
2. **工厂生命周期：** 假设 `StockFactory` 对象（比如一个全局单例或在程序生命周期内持续存在的对象）自身一直存活。
3. **无清理机制：** `get()` 方法只负责**创建并添加** `Stock` 对象到 `m_stocks`，但**没有实现任何移除或清理**的机制。

那么我们或许可以使用weak_ptr?

## version2

```c++
// version2
// 数据成员修改为std::map<string,weak_ptr<Stock>> stocks_;
// m_stocks 被修改为存储 weak_ptr<Stock>。weak_ptr 
// 不增加对象的引用计数，因此它不会阻止对象的销毁。当所有外部 shared_ptr 都释放后，对象就会自动销毁。
shared_ptr<Stock> StockFactory::get(const string& key)
{
	shared_ptr<Stock> pStock; // 返回给用户的强引用
    MutexLockGuard lock(m_mutex);
    // 如果key不存在 默认会构造一个 
    // 获取或默认构造该 key 对应的 weak_ptr 引用。
    weak_ptr<Stock>& wkStock = m_stocks[key];
    // 尝试将 weak_ptr 提升为 shared_ptr。如果对象已被销毁（引用计数为 0），提升失败，pStock 为空。
    pStock = wkStock.lock(); 
    if (!pStock) { // 如果指针为空
		pStock.reset(new Stock(key));
        wkStock = pStock;// 这里更新 m_stock[key] 注意此处wkStock是个引用
    }
    return pStock;
}
```

这么做固然Stock对象被销毁了,但是程序还是出现了轻微的内存泄漏.

当一个 `Stock` 对象被销毁后，`m_stocks` 中对应的 `weak_ptr` **并没有被移除**。它变成了一个“僵尸条目”（Zombie Entry），虽然无害，但它仍然占据了 `std::map` 中的一个键值对空间。

stocks_对象只增不减,stocks_.size() 是曾经存活过的Stock对象的总数,即便活的Stock对象数目将为0.

**后果：** 如果程序运行时间很长，并且不断创建/销毁新的 `Stock` 对象，`m_stocks` 会不断积累无效的、已销毁的 `Stock` 对应的 `weak_ptr` 条目。这将导致：

- **内存浪费：** `m_stocks` 占用的内存会持续增加（尽管增长速度很慢）。
  
- **查找效率降低：** 随着 `map` 尺寸的增加，查找时间可能会变长。

解决办法: 使用shared_ptr的定制析构功能. shared_ptr ,shared_ptr 的构造函数可以有一个额外的模板类型参数,传入一个函数指针或访问仿函数d,在析构对象执行d(ptr)

```cpp
template<class Y,class D> shared_ptr::shared_ptr(Y* p,D d);
template<class Y,class D> void shared_ptr::reset(Y* p,D d);
// Y的类型肯与T不同,这是合法的 主要Y*能够隐式转换为T*
```

那我们可以利用这一点,在析构Stock对象的同时清理stocks_

## version3

```cpp
// version3 
class StockFactroy : boost::nocopyable 
{
	// 在get()中,将pStock_reset(new Stock(key)) 改为
	// pStock.reset(newStock(key),
    //	boost::bind(&StockFactory::deleteStock,this,_1)) [1]
private:
    void deleteStock(Stock* stock)
    {
		if(stock) {
			MutexLockGuard lock(mutex_);
            stocks_.erase(stock->key);
        }
        delete stock; 
    }
    // assumeing StockFactroy lives longer than all Stock's...
	
}

```



此处存在一个严重的问题:

我们在`pStock.reset()` 中把StockFatcory保存在了boost::function里( [1] 处), 这里就会有线程安全问日

> 如果这个StockFactory 先于Stocy对象析构,那么就会core dump
> 正如Observer里的unregister [[2. 线程安全的Observer]]


此处,我使用我自己的代码来进行解释:

```cpp
    using StockPtr      = std::shared_ptr<Stock>;
    using StockWeakPtr  = std::weak_ptr<Stock>;
StockPtr get(const string& stock_key) {
        StockPtr stock_ptr;
        MutexLockGuard lock(m_mutex);
        StockWeakPtr& weak_ptr = m_stocks[stock_key];
        stock_ptr = weak_ptr.lock(); // 尝试提升为 shared_ptr
        if (!stock_ptr) {
            // 使用 shared_ptr 的自定义删除器
            stock_ptr.reset(new Stock(stock_key),
                [this, stock_key](Stock* p) {
                    // 自定义删除器，清理过期的 weak_ptr 条目
                    MutexLockGuard lock(m_mutex);
                    m_stocks.erase(stock_key);
                    delete p; // 别忘了释放对象本身
                });
            weak_ptr = stock_ptr; // 更新 weak_ptr
        }
        return stock_ptr;
    }
```

当前代码的隐患:

```cpp
stock_ptr.reset(new Stock(stock_key),
    [this, stock_key](Stock* p) {  //   ⚠️ 捕获了 this 指针
        MutexLockGuard lock(m_mutex);  // 访问 this->m_mutex
        m_stocks.erase(stock_key);      // 访问 this->m_stocks
        delete p;
    });
```

**code dump 场景**

假设按照以下时序进行:

```
时间线 ─────────────────────────────────────────────────────────►

T1: StockFactory3 factory;
T2: shared_ptr<Stock> stock = factory.get("AAPL");  // 引用计数 = 1
T3: factory 析构（离开作用域）                        // StockFactory3 对象被销毁
    ↓
    此时 m_mutex 和 m_stocks 已经是野内存！
    
T4: stock 离开作用域，引用计数 → 0，触发删除器
    ↓
    删除器执行：
    - MutexLockGuard lock(m_mutex);  ❌ 访问已销毁的 m_mutex → 未定义行为
    - m_stocks.erase(stock_key);     ❌ 访问已销毁的 m_stocks → 未定义行为
    
    💥 CORE DUMP!
```

| ------------------ | ------------------------------------------------------------ |
| **生命周期不匹配** | `Stock` 对象的生命周期可能比 `StockFactory3` 更长            |
