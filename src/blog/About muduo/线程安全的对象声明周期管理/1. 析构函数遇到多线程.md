# 析构函数与多线程

当一个对象能被多个线程同时看到时,那么对象的销毁时机就会变得模糊.可能出现多种竟态条件[race condition]

- 在即将析构一个对象的时,从何得知此刻是否有别的对象正在执行该对象的成员函数?

- 如何保证在执行成员函数期间,对象不会在另一个线程中被析构?

- 在调用某个对象的成员函数之前,如何得知这个对象是否存活? 它的析构会不会碰巧执行到一半?

# 线程安全的定义

根据[Java Concurrency in Practice],一个线程安全的class 应该满足以下条件

- **多个线程访问：** 讨论线程安全的前提是存在多线程并发访问。
    
- **无需额外同步/协调：** 这是“线程安全”的关键。用户（调用方）在使用这个对象时，不需要自己添加 `synchronized` 块、`Lock`、或者其他并发控制机制来保护它。
    
- **正确的结果：** 无论线程如何交错执行，该对象的状态始终保持一致，并且所有的操作都按照其规范正确地执行（即满足其**不变性条件**和**后置条件**）。

所以,std::string std::vector std::map等都不是线程安全的,因为这些class都需要在外部加锁才能提供多个线程同时访问

# 线程安全的Counter示例

```cpp
class Counter : public NonCopyAble {
    //copy-ctor and assignment should be privade by default for class
public:
    Counter() : m_count(0) {}

    int64_t value() const {
        // lock 的析构会晚于返回对象的构造 所以有效保护这个共享数据
        MutexLockGuard lock(m_mutex);
        return m_count;
    }
    
    int64_t getAndIncrement() {
        MutexLockGuard lock(m_mutex);
        int64_t ret = m_count++;
        return ret;
    }

private:
    int64_t m_count;
    mutable MutexLock m_mutex;
// In a real world, atomic<int64_t> m_count; would be better
};

```

# 对象的构造[easy]

对象构造要做到线程安全,唯一的要求是在构造期间不要泄漏this指针 即

- **不要在构造函数中注册任何回调**
- **不要在构造函数中把this传给跨线程的对象**
- **即使是在构造函数的最后一行也不行**
	- 因为Foo(class) 有可能是个基类,基类优先于派生类构造,执行完Foo:Foo()的最后一行代码还会继续执行派生类的构造函数,这时**most-derived class(最底层派生类[继承中位于最末端的类])**的对象还处于构造中,仍然不安全

```cpp

// Don't do this
class Foo : public Observer {
public:	
	Foo(Observable* s) {
		s->regeister_(this);  // [!!ERROR!!] 非线程安全
	}
	virtual void update();
};

// Do this.
class Foo : public Observer {
public: 
	Foo();
	virtual void update();
	
	//另外定义一个函数 在构造之后执行回调函数的注册
	void observe(Observable* s) {
		s->register_(this);
	}
	
};

Foo* p_foo = new Foo;
Observable* s = getSubject();
p_foo->observe(s); //二段式构造 或者: s->register_(pFoo);

```

# 销毁太难

### 问题的核心：析构时期的竞态条件

“销毁太难”的核心问题在于：如果一个对象正在被线程 A 析构，而同时线程 B 仍然在使用或即将使用该对象内部的某个同步原语，就会导致**未定义行为**和程序崩溃。

#### 1. 析构函数的行为

在 C++ 中，析构函数的目的是**释放资源**。对于互斥锁（如 `pthread_mutex_t`），这通常意味着调用一个销毁函数（如 `pthread_mutex_destroy`）来释放底层操作系统资源。

#### 2. 发生冲突的场景（销毁 vs. 使用）

假设有一个包含互斥锁 `m_mutex` 的对象 `LockObject`。

|**线程 A (析构者)**|**线程 B (使用者)**|**结果**|
|---|---|---|
|调用 `~LockObject()` 析构函数。|线程 B 正在某个成员函数中持有 `LockObject` 的锁。|**未定义行为：** 线程 A 试图销毁一个正在被线程 B 使用的锁。|
|调用 `pthread_mutex_destroy(&m_mutex)`。|线程 B 紧接着尝试调用 `pthread_mutex_unlock(&m_mutex)`。|**崩溃/未定义行为：** `m_mutex` 已经被销毁。线程 B 试图操作一个**已销毁的同步原语**。|

即互斥器本身必须是有效的），它会把 `mutex` 成员变量销毁掉。悲剧啊！”

---

###  muduo 库如何解决“销毁太难”

`MutexLock` 代码（仿 muduo）通过引入一个强有力的**断言（Assertion）**机制，从设计上强制避免了这种危险的竞态：

#### 关键机制：析构断言

C++

```cpp
~MutexLock()
{
    assert(holder_ == 0);
    pthread_mutex_destroy(&mutex_);
}
```

1. **检查持有者：** 析构函数的第一步是使用 `assert(holder_ == 0)` 来检查当前是否**有任何线程持有**这把锁。
    
    - `holder_` 记录了锁的持有者 TID。`holder_ == 0` 表示锁是空闲状态。
        
2. **强制安全：**
    
    - **如果断言成功：** `holder_` 等于 0，这意味着锁处于未被任何线程使用的安全状态，可以继续调用 `pthread_mutex_destroy` 进行安全销毁。
        
    - **如果断言失败：** `holder_` 不等于 0，这表明**某个线程仍在临界区内**，或者忘记释放锁。**这是一个严重的并发设计错误。** 在调试模式下，`assert` 会立即终止程序，防止继续执行危险的 `pthread_mutex_destroy` 操作。
        

通过这种设计，`MutexLock` 强制要求使用者在对象析构之前，**必须**确保所有线程都已经退出临界区并释放了锁。它将运行时可能发生的崩溃，转化为开发阶段就能发现并修复的**设计错误**。

# 作为数据成员的mutex 不能保护析构

代码如下

```cpp
class Foo;
Foo::~Foo() {
	MutexLockGuard lock(m_mtx);
	//(1) 
} 

void Foo::update() {
	MutexLockGuard lock(m_mtx);
	// (2)
}

```

此时有A,B 两个线程可以访问Foo对象 x 
线程A即将销毁x, 而线程B 正准备调用x->update();

```cpp
extern Foo* x; // visiable by all threads

//thread A

    delete x;
    x = nullptr;


// thread B

    if(x) {
        x->update();
    }

//尽管 threadA 在销毁对象之后把指针设为 nullptr
//尽管 threadB 在调用x->update()之前检查了x是否为nullptr
//但仍然存在竞态
//可能的执行顺序如下：
//1. 线程A执行到了析构函数的 MutexLockGuard lock(m_mtx); 处，成功持有锁 即将继续往下执行
//2. 线程B通过了 if(x) 检查,阻塞在了 MutexLockGuard lock(m_mtx); 处，等待锁
//3. 线程A继续执行，完成析构函数，释放锁
//4. 线程B获得锁，继续执行x->update()，此时x已经是一个悬空指针，导致未定义行为

```

说明,delete对象之后把指针置为NULL根本没用. 

作为class数据成员的MutexLock只能用于同步本地class的其他数据成员的读写,它不能保护安全的析构,因为MutexLock的成员的生命周期最多和对象一样长. 

另外如果要同时读写一个class的两个对象,有潜在死锁可能 比如:

```cpp
class Counter {

public:
    Counter& operator=(Counter& rhs);
    int64_t m_value;
    MutexLock m_mutex;
};

void swap(Counter& a, Counter& b) {
    MutexLockGuard lock_a(a.m_mutex);
    MutexLockGuard lock_b(b.m_mutex);
    int64_t temp = a.m_value;
    a.m_value    = b.m_value;
    b.m_value    = temp;
}

// 这个swap函数存在死锁风险
// 假设有两个线程T1和T2同时调用swap函数，传入的参数顺序相反：
// 线程T1调用swap(counterA, counterB)
// 线程T2调用swap(counterB, counterA)
// 可能的执行顺序如下：
// 1. 线程T1执行到 MutexLockGuard lock_a(a.m_mutex); 成功获得 counterA 的锁
// 2. 线程T2执行到 MutexLockGuard lock_a(a.m_mutex); 成功获得 counterB 的锁
// 3. 线程T1继续执行，尝试获得 counterB 的锁，但此时锁已被线程T2持有，T1阻塞等待
// 4. 线程T2继续执行，尝试获得 counterA 的锁，但此时锁已被线程T1持有，T2阻塞等待
// 结果：线程T1等待线程T2释放 counterB 的锁，线程T2等待线程T1释放 counterA 的锁，形成死锁

Counter& Counter::operator=(Counter& rhs) {
    if (this == &rhs) {
        return *this;
    }
    MutexLockGuard this_lock(this->m_mutex);
    MutexLockGuard rhs_lock(rhs.m_mutex);
    this->m_value = rhs.m_value;
    return *this;
}
// 这个赋值操作符重载函数同样存在死锁风险
// 假设有两个线程T1和T2同时调用赋值操作符，
// 传入的参数顺序相反：
// 线程T1执行 counterA = counterB;
// 线程T2执行 counterB = counterA;
// 可能的执行顺序如下：
// 1. 线程T1执行到 MutexLockGuard this_lock(this->m_mutex); 成功获得 counterA 的锁
// 2. 线程T2执行到 MutexLockGuard this_lock(this->m_mutex); 成功获得 counterB 的锁
// 3. 线程T1继续执行，尝试获得 counterB 的锁，但此时锁已被线程T2持有，T1阻塞等待
// 4. 线程T2继续执行，尝试获得 counterA 的锁，但此时锁已被线程T1持有，T2阻塞等待
// 结果：线程T1等待线程T2释放 counterB 的锁，线程T2等待线程T1释放 counterA 的锁，形成死锁
```

**所以:**
**一个函数如果要锁住相同类型的多个对象,为了保证始终按相同的的顺序加锁,我们可以比较mutex对象的地址,始终先加锁地址较小的mutex**

```cpp
// 解决方法：比较mutex地址，始终给较小地址的mutex加锁
void safe_swap(Counter& a, Counter& b) {
    if (&a == &b) {
        return;
    }
    Counter* first  = &a;
    Counter* second = &b;
    if (std::addressof(a) > std::addressof(b)) {
        first  = &b;
        second = &a;
    }
    MutexLockGuard lock_first(first->m_mutex);
    MutexLockGuard lock_second(second->m_mutex);
    int64_t temp = a.m_value;
    a.m_value    = b.m_value;
    b.m_value    = temp;
}
```

```cpp
// C++11/C++14/C++17 的标准库解决方案
void safe_swap_std(Counter& a, Counter& b) {
    // std::lock 会同时锁定 a.m_mutex 和 b.m_mutex，
    // 自动以无死锁的方式（如内部使用地址或尝试失败机制）进行加锁。
    std::lock(a.m_mutex, b.m_mutex);
    
    // std::lock_guard 绑定已锁定的 mutex，并在作用域结束时自动解锁。
    // std::adopt_lock 标记告知 lock_guard 锁已经被外部持有。
    std::lock_guard<std::mutex> lock_a(a.m_mutex, std::adopt_lock);
    std::lock_guard<std::mutex> lock_b(b.m_mutex, std::adopt_lock);
    
    // 执行交换操作...
    int64_t temp = a.m_value;
    a.m_value    = b.m_value;
    b.m_value    = temp;
}
```
