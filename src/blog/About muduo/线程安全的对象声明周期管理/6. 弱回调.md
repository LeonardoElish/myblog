# 弱回调

[[../5. enable_shared_from_this|5. enable_shared_from_this]] 上文

把`shared_ptr`绑定到(std::bind)到(std::function)里,那么回调的时候`StockFactroy`对象始终是存在的,是安全的.这同时也延长了对象的生命期,使之不短于 绑定的`std::function`对象.


在 **lambda 函数**中，如果正确地将一个 **`std::shared_ptr`** **捕获**到其闭包中，那么这个 `std::shared_ptr` 所管理的对象（如您例子中的 `StockFactory` 对象）的生命周期**也会**得到延长，使其不短于该 lambda 函数对象本身的生命周期。

## **lambda 函数中的机制**

- **捕获 `std::shared_ptr`**：
  
    - 当您通过**值捕获**（`[ptr]` 或 `[=]`，如果 `ptr` 在作用域内）的方式将一个 `std::shared_ptr` 实例捕获到 lambda 闭包中时，lambda 对象会存储一个该 `shared_ptr` 的**拷贝**。
      
    - 这个拷贝会**增加** `shared_ptr` 的**引用计数**。
      
    - 只要 lambda 函数对象本身存在（例如，它被存储在一个 `std::function` 对象中），这个 `shared_ptr` 的拷贝就会一直存在，从而保证它所指向的底层对象（`StockFactory`）不会被销毁。
    
- **安全性和生命周期延长**：
  
    - 这与您在 `std::bind` 中使用 `std::shared_ptr` 作为参数或绑定对象的效果是**一致的**。
      
    - 在回调执行时，由于 `shared_ptr` 仍持有对对象的引用，所以 `StockFactory` 对象是**始终存在且安全的**。

## 弱回调机制

有时候我们需要 "如果对象还活着,就调用它的成员函数否者忽略之". 就像`Observer::notifyObservers`那样([[2. 线程安全的Observer]]). 这称之为弱回调

我们可以利用`weak_ptr`,我们可以把`weak_ptr`绑定到`std::function`里,这样对象的生命周期就不会被延长,然后在回调的时候先尝试提升为`shared_ptr`,如果提升成功,就说明接受回调的对象还存货,那么就执行回调,如果提升失败,就不必劳神了

完整代码如下:
```cpp
class StockFactory5 : public std::enable_shared_from_this<StockFactory5>
{
    using StockPtr          = std::shared_ptr<Stock>;
    using StockWeakPtr      = std::weak_ptr<Stock>;
    using WkStockFactoryPtr = std::weak_ptr<StockFactory5>;
    using StockFactoryPtr   = std::shared_ptr<StockFactory5>;
public:
    StockPtr get(const string& stock_key) {
        StockPtr stock_ptr;
        {
            MutexLockGuard lock(m_mutex);
            StockWeakPtr& weak_ptr = m_stocks[stock_key];
            stock_ptr = weak_ptr.lock(); // 尝试提升为 shared_ptr
            if (!stock_ptr) {
                WkStockFactoryPtr weak_factory = shared_from_this();
                stock_ptr.reset(new Stock(stock_key),
                    std::bind(&StockFactory5::weakDeleteCallback,
                              weak_factory,
                              std::placeholders::_1));
                weak_ptr = stock_ptr; // 更新 weak_ptr
            }
        }
        return stock_ptr;
    }
private:

    void removeStock(Stock* p) {
        if (p) {
            MutexLockGuard lock(m_mutex);
            m_stocks.erase(p->getKey());
        }
    }

    static void weakDeleteCallback(const WkStockFactoryPtr& weak_factory,                                   
                                   Stock* p) {
        StockFactoryPtr factory(weak_factory.lock());
        if (factory) {
            factory->removeStock(p);
        }
        delete p; // sorry i lied;
    }

    mutable MutexLock m_mutex;
    std::map<std::string, StockWeakPtr> m_stocks;
};
```

output:

```bash

Stock1 key: NYSE:IBM
Stock2 key: NYSE:IBM
Stock destructor: NYSE:IBM
StockFactory5 destructor
StockFactory5 constructor
Stock constructor: NYSE:IBM
Stock1 key: NYSE:IBM
StockFactory5 destructor
Stock destructor: NYSE:IBM
```

这下完美了,无论`Stock`和`StockFactory`谁先析构都不会影响程序的正确运行.
此处我们借助了shared_ptr和weak_ptr完美地解决了两个对象相互引用的问题.

当然,通常Factory对象是个Singleton,在程序运行期间不会销毁,此处只是为了展示弱回调技术,这个技术在事件通知中非常有用
