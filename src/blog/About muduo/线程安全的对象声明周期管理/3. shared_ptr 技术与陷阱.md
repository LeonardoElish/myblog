# shared_ptr技术与陷阱

# bind陷阱

在`boost::bind` 和 `std::bind` 中,在默认情况下会对所有传递给它的**实参**进行**值拷贝（copy）**。

- **`shared_ptr` 的行为：** 当 `shared_ptr` 被拷贝时，它的**引用计数**会增加

```cpp
shared_ptr<Foo> pFoo(new Foo);
boost::function<void()> func = boost::bind(&Foo::doit, pFoo);
```

`boost::bind` 将 `pFoo` **拷贝**了一份，这份拷贝被存储在了最终的 `boost::function` 对象 `func` 内部。只要 `func` 对象存在，它内部的 `shared_ptr` 拷贝就存在，`Foo` 对象的引用计数就至少为 1，因此 `Foo` 对象就不会被销毁，从而**延长了它的生命周期*

std::bind

```cpp
#include <iostream>
#include <functional>
#include <memory>

class Foo {
public:
    void doit() { std::cout << "Foo::doit called." << std::endl; }
    ~Foo() { std::cout << "Foo destroyed." << std::endl; }
};

void run_std_bind_issue() {
    std::cout << "--- std::bind 演示 ---\n";
    
    // 1. 创建原始 shared_ptr (引用计数 = 1)
    std::shared_ptr<Foo> pFoo(new Foo);
    std::cout << "原始 pFoo 引用计数: " << pFoo.use_count() << "\n"; // 1

    // 2. 使用 std::bind 绑定成员函数和对象实例
    // std::bind 会拷贝 pFoo，将其存储在新函数对象 f 内部
    auto f = std::bind(&Foo::doit, pFoo); 
    std::cout << "绑定后 pFoo 引用计数: " << pFoo.use_count() << "\n"; // 2 (原始 pFoo + f 内部的拷贝)

    // 原始 pFoo 离开作用域，引用计数 -1 (变为 1)
} // f 离开作用域，f 内部的 shared_ptr 销毁，引用计数 -1 (变为 0)，Foo 对象被销毁。

// 演示函数结束时，Foo 对象才会被销毁

```

## 函数传参

因为要修改引用计数(而且拷贝的时候通常需要加锁), shared_ptr的拷贝开销比拷贝原始指针要高, 多数情况下可以以 const reference 进行传递.

一个线程只需要在最外层函数中有一个实体对象, 之后都可以以用const reference 来使用这个shared_ptr 

```cpp

void save(const shared_ptr<Foo>& pFoo); // pass by const reference
void validateAccount(const Foo& foo);

bool validate(const shard_ptr<Foo>& pFoo){
	validateAccount(*pFoo);
}

```

通常情况下,我们可以传常引用

```cpp
void onMessage(const string& msg) {
	shared_ptr<Foo> pFoo(new Foo(msg)); // 只需要在最外层持有一个实体 安全不成问题
	if(validate(pFoo)) { // no cpoy pFoo
		save(pFoo);
	}
} 

```

析构函数在创建时被捕获 这意味着:

- 虚析构不再必须
- shared_ptr\<void\> 可以持有任何对象,而且能安全的释放
- shared_ptr 对象可以安全的跨越模块边界,比如从DLL里返回,而不会造成从模块A分配的内存在模块B里被释放的这种错误
- 二进制兼容性
	- 即使Foo对象的大小变了,那么旧的客户代码仍然可以使用新的动态库而无需重新编译(前提是Foo的头文件不出现访问对象的成员的inline函数,并且Foo对象的动态库中的Factory构造 返回其shared_ptr)
- 析构动作可定制



**析构所在的线程:**

> 对象的析构是同步的,当最后一个指向`x`的`shared_ptr`离开其作用域的时候,x会同时在**同一个**线程析构. 这个线程不一定是对象诞生的线程
>
> 如果对象的析构比较耗时,那么可能会拖慢关键线程的速度(如果最后一个`shared_ptr `引发的析构发生在关键线程); 同时,我们可以用一个单独的线程来专门做析构, 通过一个`BlockingQueue<shared_ptr<void>>`把对象的析构都转移到那个专用线程 从而解放关键线程 详细见下文



`shared_ptr` 需要避免循环引用, 通常做法是owner持有指向child的`shared_ptr`,child 持有指向owner的`weak_ptr`





##  异步析构器与类型擦除

```cpp
#include <memory>
#include <thread>
#include <mutex>
#include <condition_variable>
#include <queue>

// 阻塞队列实现
template<typename T>
class BlockingQueue {
public:
    void put(T&& item) {
        std::lock_guard<std::mutex> lock(mutex_);
        queue_.push(std::move(item));
        cond_.notify_one();
    }
    
    /**
     * @brief 阻塞地等待队列中出现元素，然后安全地取出并返回该元素。
     */
    T take() {
        std::unique_lock<std::mutex> lock(mutex_);
        cond_.wait(lock, [this] { return !queue_.empty() || stopped_; });
        if (stopped_ && queue_.empty()) {
            return nullptr;
        }
        T item = std::move(queue_.front());
        queue_.pop();
        return item;
    }
    
    void stop() {
        std::lock_guard<std::mutex> lock(mutex_);
        stopped_ = true;
        cond_.notify_all();
    }
    
private:
    std::queue<T> queue_;
    std::mutex mutex_;
    std::condition_variable cond_;
    bool stopped_ = false;
};

// 异步析构器 
/**
 * @brief 解决由于对象析构耗时过长而可能导致的关键线程（如主线程、网络 I/O 线程）阻塞的问题。
 * *      AsyncDestructor 类利用一个专用的后台线程和一个阻塞队列，
 *        将对象的销毁操作从调用线程（例如正在运行业务逻辑的主线程）转移到这个后台线程中执行，
 *        从而保证调用线程的执行流畅性。
*/
class AsyncDestructor {
public:
    AsyncDestructor() : thread_(
        [this] { destructorLoop(); }
    ) {}
    
    ~AsyncDestructor() {
        
        queue_.stop();  // 告知阻塞队列停止接受新任务，并唤醒所有正在 take() 上等待的线程（即 destructorLoop）。
        thread_.join(); // 等待后台析构线程执行完队列中剩余的所有任务后，
                        // 安全地退出循环并终止。这确保了在 AsyncDestructor 对象本身被销毁之前，
                        // 所有待销毁的对象都已清理完毕。
    }
    
    // 将对象转移到析构线程
    // 使用 shared_ptr<void> 利用类型擦除，可以接受任意类型
    // 调用时机： 当关键线程决定一个对象（由 ptr 指向）不再需要时，调用此方法。
    // std::move(ptr)： 将 shared_ptr<T> 的所有权转移到临时对象中。调用线程不再持有此 shared_ptr，其引用计数会递减。
    // 类型转换： std::shared_ptr<void>(...) 将 shared_ptr<T>
    //  安全转换为 shared_ptr<void>，同时保留正确的控制块（包括析构信息）。
    template<typename T>
    void destroy(std::shared_ptr<T>& ptr) {
        // 将 shared_ptr<T> 转为 shared_ptr<void>
        // 析构时仍会调用正确的析构函数（因为 deleter 被保存了）
        // 将这个包含对象析构任务的 shared_ptr<void> 放入队列。析构任务被成功转移到后台线程。
        queue_.put(std::shared_ptr<void>(std::move(ptr))); 
    }
    
private:
    void destructorLoop() {
        // 阻塞的从队列取出取出下一个 shared_ptr<void> 对象（即 obj）。如果队列为空，线程会休眠等待。
        while (auto obj = queue_.take()) {
            // obj 在这里离开作用域时自动析构
            // 析构发生在这个专用线程，而非关键线程
        }
    }
    
    // 线程安全阻塞队列 负责存储待销毁的对象
    // std::shared_ptr： 
    // 用于保证待销毁对象的生命周期。只要 shared_ptr 仍在队列中或被 destructorLoop 持有，对象就不会被销毁。
    // void：
    // 这是一个关键设计。shared_ptr<void> 是一种类型擦除（Type Erasure）的技巧。
    // 它可以持有指向任何类型 T 的指针，但最重要的是，它依然保留了正确的析构器（Deleter）信息。
    // 无论放入什么类型的对象，当这个 shared_ptr<void> 最终被销毁时，它都能调用正确的析构函数 ~T()。
    BlockingQueue<std::shared_ptr<void>> queue_;

    // 专门用于执行对象析构操作的后台线程。
    std::thread thread_;
};

// 全局析构器实例
AsyncDestructor g_asyncDestructor;

// ============ 使用示例 ============

class HeavyObject {
public:
    HeavyObject(int id) : id_(id) {
        printf("[Thread %zu] HeavyObject %d constructed\n", 
               std::hash<std::thread::id>{}(std::this_thread::get_id()) % 1000, id_);
    }
    
    ~HeavyObject() {
        // 模拟耗时析构
        std::this_thread::sleep_for(std::chrono::milliseconds(100));
        printf("[Thread %zu] HeavyObject %d destructed (slow!)\n",
               std::hash<std::thread::id>{}(std::this_thread::get_id()) % 1000, id_);
    }
    
    void doWork() { printf("HeavyObject %d working\n", id_); }
    
private:
    int id_;
};

// 关键线程中的使用
void criticalThreadWork() {
    printf("=== Critical thread start ===\n");
    
    auto obj = std::make_shared<HeavyObject>(1);
    obj->doWork();
    
    // 方式1: 直接让 shared_ptr 析构 (会阻塞关键线程)
    // obj.reset();  // 不推荐！析构在当前线程进行
    
    // 方式2: 转移到异步析构线程 (关键线程立即返回)
    g_asyncDestructor.destroy(obj);  // 推荐！
    
    // 此时 obj 已经是 nullptr，析构将在专用线程进行
    printf("=== Critical thread done (no blocking!) ===\n");
}

int main() {
    criticalThreadWork();
    
    // 主线程继续其他工作...
    printf("Main thread continues...\n");
    std::this_thread::sleep_for(std::chrono::milliseconds(200));
    
    return 0;
}
```
