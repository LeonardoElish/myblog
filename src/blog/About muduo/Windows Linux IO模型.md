五种模型分别是：

1. **阻塞 I/O (Blocking I/O)**
    
2. **非阻塞 I/O (Non-blocking I/O)**
    
3. **I/O 复用/多路转接 (I/O Multiplexing)** (如 `select`, `poll`, `epoll`)
    
4. **信号驱动 I/O (Signal Driven I/O)**
    
5. **异步 I/O (Asynchronous I/O)**
    

## 1. 阻塞 I/O (Blocking I/O)

### 概念

这是最传统的 I/O 模型。当应用程序调用 `recvfrom`（或其他 I/O 函数）时，进程会被**阻塞**。它会一直等到数据报到达且被复制到应用程序缓冲区中，或者发生错误，才返回。

### 过程

1. **等待数据阶段 (Wait for Data)**: 进程阻塞，等待数据准备好。
    
2. **复制数据阶段 (Copy Data)**: 数据从内核缓冲区复制到用户缓冲区。
    
3. **返回**: 函数成功返回。
    

### 适用场景

简单、低并发的程序。

---

## 2. 非阻塞 I/O (Non-blocking I/O)

### 概念

应用程序将套接字设置为非阻塞模式。当应用程序调用 I/O 函数时，如果数据还没有准备好，函数会**立即返回一个错误**（例如 Linux 下的 `EWOULDBLOCK` 或 `EAGAIN`），而不是阻塞进程。应用程序需要通过**轮询 (Polling)** 的方式，不断调用 I/O 函数来检查数据是否准备好。

### 过程

1. **等待数据阶段**: 进程不断调用 I/O 函数，每次调用都**立即返回**，直到数据准备好。
    
2. **复制数据阶段**: 数据从内核缓冲区复制到用户缓冲区。
    
3. **返回**: 函数成功返回。
    

### 适用场景

需要处理少量并发连接，但不希望主程序被阻塞的场景。但频繁的系统调用会消耗 CPU 资源。

---

## 3. I/O 复用/多路转接 (I/O Multiplexing)

### 概念

**I/O 复用**允许一个进程监视多个文件描述符（Socket），一旦其中任何一个文件描述符上有 I/O 事件发生（如可读或可写），**`select`/`poll`/`epoll`** 就会返回。进程随后调用实际的 I/O 函数（如 `read`）来处理数据。

### 过程

1. **调用复用函数**: 进程调用 `select`/`poll`/`epoll`，进程**阻塞**在这一步。
    
2. **等待事件**: 内核等待被监视的描述符上有事件发生。
    
3. **复用函数返回**: 当描述符就绪时，复用函数返回。
    
4. **调用 I/O 函数**: 进程调用 `read` 等 I/O 函数，数据从内核复制到用户空间（这一步通常是**阻塞**的，但因为之前已经确认数据就绪，所以阻塞时间极短）。
    

### 关键点

- **模型归类**: 尽管可以同时处理多个 I/O，但从本质上讲，I/O 复用是**同步 I/O** 的一种形式，因为数据复制阶段（步骤 4）进程仍然需要等待完成。
    
- **主流实现**:
    
    - **`select`**: 效率最低，有最大文件描述符限制。
        
    - **`poll`**: 解决了文件描述符限制，但效率仍一般。
        
    - **`epoll` (Linux)** / **`IOCP` (Windows)** / **`kqueue` (FreeBSD/macOS)**: 效率最高，使用事件驱动机制，是高并发服务器的主流选择。
        

### 适用场景

高并发、单进程服务器架构（如 muduo, Nginx, Redis）。

---

## 4. 信号驱动 I/O (Signal Driven I/O)

### 概念

应用程序可以向内核注册一个信号处理程序（例如使用 `sigaction`），并设置文件描述符启用信号驱动 I/O（如 Linux 下的 `SIGIO` 信号）。当数据准备好时，内核会发送一个信号通知进程。进程可以继续执行其他任务，在收到信号后，再调用 `recvfrom` 来处理数据。

### 过程

1. **注册信号**: 进程注册信号处理函数，并立即返回，**不阻塞**。
    
2. **等待信号**: 进程继续执行其他任务。
    
3. **信号通知**: 数据准备好时，内核发送信号。
    
4. **调用 I/O 函数**: 进程收到信号，在信号处理函数或主循环中调用 `read` 等函数，数据从内核复制到用户空间（这一步**阻塞**）。
    

### 适用场景

较少使用，因为信号处理机制比较复杂。它与 I/O 复用相比，优势在于**等待数据阶段**完全是非阻塞的。

---

## 5. 异步 I/O (Asynchronous I/O)

### 概念

**异步 I/O (AIO)** 是最高级的 I/O 模型。应用程序调用 I/O 函数后，**立即返回**，进程可以继续执行其他任务。**整个 I/O 操作**（包括等待数据和将数据从内核复制到用户空间）完成后，内核会通知应用程序。

### 过程

1. **调用 AIO 函数**: 进程调用 `aio_read` 等函数，**立即返回**。
    
2. **内核处理**: 内核开始处理 I/O，包括等待数据和数据复制。
    
3. **通知完成**: **整个 I/O 过程**完成后，内核向进程发送通知（如信号或完成例程）。
    

### 关键点

- **与同步 I/O 的根本区别**:
    
    - **同步 I/O**：I/O 操作的第二阶段（数据复制）是阻塞的。
        
    - **异步 I/O**：I/O 操作的两个阶段（等待数据和数据复制）都是非阻塞的。
        

### 适用场景

需要极高性能 I/O 并发的场景，例如数据库服务器或高性能文件系统。

### 总结比较

|**模型**|**I/O 等待 (Phase 1)**|**数据复制 (Phase 2)**|**进程状态**|**核心函数 (Linux)**|
|---|---|---|---|---|
|**阻塞 I/O**|**阻塞**|**阻塞**|进程挂起|`recvfrom`|
|**非阻塞 I/O**|**非阻塞 (轮询)**|**阻塞**|进程忙等/轮询|`recvfrom` (O_NONBLOCK)|
|**I/O 复用**|**阻塞**|**阻塞**|进程挂起 (在 `select/epoll` 上)|`select`/`epoll_wait`|
|**信号驱动 I/O**|**非阻塞**|**阻塞**|进程继续执行|`sigaction` (SIGIO)|
|**异步 I/O**|**非阻塞**|**非阻塞**|进程继续执行|`aio_read`|
