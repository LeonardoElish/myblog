# 基本原则
并发编程两种基本模型
- message passing
- shared memeory

在分布式系统中，运行在多台机器多个进程的并发编程只有“shared memory”

线程同步的四项原则（按照重要性排列：
1. 首要原则是最低限度地共享对象，减少需要同步的场合。一个对象能暴露给别的线程就不要暴露，如果要暴露，优先考虑immutable（不可改变）对象；是在不行才暴露可修改对象，并用同步措施来充分保护它
2. 其次是使用高级的并发编程构件。 如：TaskQueue、Producter-Consumer Queue、CountDownLatch
3. 最后不得已才使用底层同步原语（primitives）时，之使用非递归的互斥器和条件变量、慎用读写锁、不要用信号量。
4. 除了使用atomic整数之外，不自己编写lock-free代码，也不要用内核级同步语句。不凭空猜测“哪种做法性能会更好”，比如spin lock vs mutex


# 互斥器（mutex

单独使用mutex时，我们主要为了保护共享数据，原则如下

- 用RAII手法封装mutex的创建、加锁、销毁、解锁这四个步骤。保证锁的生效期等于一个作用域（scope），不会因为异常而忘记解锁
- 使用非递归的mutex（即不可重入的mutex）non-recursive mutex
	- std::mutex (default non-recursive mutex)
	- pthread_mutex_t (default non-recursive mutex)
		- `pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER;`
	- `SRWLOCK lock; InitializeSRWLock(&lock);` 
	- `CRITICAL_SECTION cs`; !!!(default recursive mutex)
- 不手动调用lock() unlock()函数，一切都交给栈上的Guard对象的构造和析构函数负责。Guard对象的生命周期正好等于临界区 。 锁的获取和释放必须在同一作用域内完成，不要跨函数、不要跨逻辑分支。
- 在每次构造Guard对象的时候，思考调用栈上已经持有的锁，防止因为加锁顺序不同而导致的死锁（deadlock）。
- 不使用跨进程的mutex，进程间通信之使用Tcp socket
- 加锁、解锁在同一个线程	
- 别忘了加锁解锁（RAII 自动保证）
- 必要时可考虑吧使用`PTHREAD_MUTEX_ERRORCHECK`来进行排错

# 只使用非递归的mutex

mutex分为：
	- 递归（recursive）可重入的
	- 非递归（non- recursive）不可重入的
唯一区别：
	同一个线程可以重复对recursive mutex加锁，但是不能重复对non-recursive加锁

递归锁和非递归锁存在设计意图上的差异。在同一个线程里多次对non-recursive mutex加锁会立刻死锁。这能帮助我们在编码阶段便发现问题。

毫无疑问recursive mutex是更方便的，但这也隐藏了代码里的一些问题. 典型情况如下：
详细代码见：`blog/About muduo/线程同步精要/ch2/NonRecursiveMutexTest.cc`

```bash
 post() 加锁修改了共享资源 foos, traverse() 加锁遍历了共享资源 foos
 这都是没问题的。但是如果如下操作：
 Foo::doit() 中调用 post()，就会死锁。
 因为 post() 和 traverse() 都需要获取 mutex 锁，
 而 post() 在调用 traverse() 时已经持有了 mutex 锁，所以 traverse() 无法获取到 mutex锁而导致死锁

 1. mutex 是非递归锁，不能被同一线程多次加锁。
 2. mutex 是递归锁 （如 pthread_mutex_t 的 PTHREAD_MUTEX_RECURSIVE 类型
    由于push_back() 可能会导致 vector 重新分配内存，导致迭代器失效程序崩溃，所以 post() 和 traverse() 不能同时持有 mutex 锁。
```

```c++

MutexLock mutex;
std::vector<Foo> foos;

void post(const Foo& f)
{
  MutexLockGuard lock(mutex);
  foos.push_back(f);
}

void traverse()  // 线程安全的遍历foos
{
  MutexLockGuard lock(mutex);
  for (std::vector<Foo>::const_iterator it = foos.begin();
      it != foos.end(); ++it)
  {
    it->doit();
  }
}

void Foo::doit() const
{
  Foo f;
  post(f);
}

int main()
{
  Foo f;
  post(f);
  traverse();
}
```

这时候就能体现出non-recursive-mutex的优越性，能把代码的逻辑错误暴露出来，死锁比较容易debug
把各个程序的调用栈打印出来，很容易看出是怎么死锁的（可以使用PTHREAD_MUTEX_ERRORMUTEX）

如果确实需要在遍历vector的时候修改vector，有以下两种做法
1. 将修改推后，记住循环中试图添加或删除的哪些元素，等循环结束再依照记录修改vector
2. 使用copy-on-write
	example: blog/About muduo/线程安全的对象声明周期管理/ch1/SignalSlot.hpp

如果一个函数即可能在已加锁的情况下调用，有可能在未加锁的情况下调用。那么应该拆分为两个函数：
1. 跟原来的函数同名，函数加锁，转为调用第二个函数
2. 给函数名后缀加上withLockHold，不加锁把原来的函数体搬过来

```c++
void post(const Foo&f) {
	MutexLockGuard mutex(lock);
	postWithLockHold();// 不用担心性能开销 编译器会自动内联
}

void postWithLockHold(const Foo& f){
	foo.push_back(f);
}

/// 然而这样可能导致两个问题
/// a.误用了加锁版本死锁了
/// b.误用了不加锁版本，数据损坏了 ： 详情见代码部分`NonRecursiveMutexTest.cc`
```

对于b 如果thread提供了`isLockByThisThread()`就可以如下解决
```c
void postWithLockHold(const Foo&f) {
	assert(mutex.isLockByThisThread()); // 运行时检查是否持有锁
}
```
关于锁的性能：
Linux的PTHREAD Mutex 采用futex(2)实现，不必每次加锁、解锁都陷入系统调用，效率不错
Windows的CRITICAL_SECTION也是类似的，不过他可以加入一小段的spin lock
在多CPU系统上，如果不能立刻拿到锁，他会先spin一小段时间，如果还不能拿到锁，才挂起当前线程
